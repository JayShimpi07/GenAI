Introduction to LangChain

Large Language Models are impressive in isolation — they can write, reason, and summarize with remarkable fluency. But raw LLM calls are barely scratching the surface of what's possible. Real applications need to remember past conversations, call external tools, search private documents, and chain multiple reasoning steps together. Wiring all of this from scratch is a non-trivial engineering problem.

LangChain is an open-source Python (and JavaScript) framework that solves exactly this. It provides modular, composable abstractions — from simple prompt templates to full autonomous agents — that let you build production-grade LLM-powered applications without reinventing the wheel every time.

What LangChain solves
LangChain handles three hard problems: Orchestration (connecting LLMs to data and actions), Chaining (routing output of one step as input to the next), and Tool Integration (letting the LLM call APIs, search engines, databases, and more).

Before LangChain, developers hand-rolled fragile glue code — manual prompt formatting, ad-hoc history concatenation, and brittle conditional logic around tool calls. LangChain replaces all of this with a clean, extensible interface. It integrates with OpenAI, Anthropic, HuggingFace, Google, Cohere, and dozens of vector databases out of the box.

Core Components of LangChain

LangChain is best understood as a set of composable building blocks. Each component has a single clear responsibility and can be combined with others to form pipelines of arbitrary complexity.

🧠

LLMs & Chat Models

Unified interface to any language model — OpenAI, Anthropic, HuggingFace, local models.

📝

Prompt Templates

Parameterized templates with variables, few-shot examples, and structured formatting.

⛓️

Chains

Pipelines that compose multiple components — prompt → LLM → parser → another prompt.

🧩

Memory

Pluggable memory backends (buffer, summary, entity, vector) for persistent conversation state.

🤖

Agents

Reasoning loops where the LLM decides which tool to use, calls it, and iterates toward a goal.

🔧

Tools

Any function the agent can call — web search, calculator, SQL, custom APIs, Python executor.

📄

Document Loaders

Ingest PDFs, web pages, Notion, GitHub repos, databases, and 50+ other sources.

🗄️

Vector Stores

Embed and index documents; retrieve semantically relevant chunks using cosine similarity.

LLMs and Chat Models

LangChain wraps any model behind a common interface. ChatOpenAI is the most common entry point. The key distinction is between completion models (plain text in/out) and chat models (structured message lists with roles). Modern applications almost always use chat models.

Python
# llm_basics.py — Basic LLM call
from langchain_openai import ChatOpenAI
from langchain_core.messages import HumanMessage, SystemMessage

# Initialize model — temperature=0 for deterministic outputs
llm = ChatOpenAI(
    model="gpt-4o-mini",
    temperature=0,
    max_tokens=512
)

messages = [
    SystemMessage(content="You are a concise data science tutor."),
    HumanMessage(content="Explain gradient descent in 2 sentences.")
]

response = llm.invoke(messages)
print(response.content)
# → "Gradient descent is an optimization algorithm that iteratively adjusts
#    model parameters by moving in the direction of steepest descent of the
#    loss function. Each update is scaled by the learning rate."

Prompts and Prompt Templates

Hard-coding strings into your LLM calls is fragile. ChatPromptTemplate lets you define reusable, parameterized message templates. Variables are injected at runtime, keeping your logic DRY and testable.

Python
# prompt_templates.py
from langchain_core.prompts import ChatPromptTemplate

template = ChatPromptTemplate.from_messages([
    ("system", "You are an expert in {domain}. Be concise and technical."),
    ("human",  "Explain: {concept}")
])

# Format fills in variables before sending to the model
formatted = template.format_messages(
    domain="machine learning",
    concept="the attention mechanism"
)
print(formatted[1].content)
# → "Explain: the attention mechanism"

Chains — The LCEL Revolution

A Chain is a pipeline that links components together. LangChain's modern API uses LangChain Expression Language (LCEL) — the pipe operator (|) composes components lazily, enabling streaming, parallel execution, and easy debugging.

Python
# chains.py — Building chains with LCEL
from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser

prompt = ChatPromptTemplate.from_template(
    "Summarize the following in one sentence:\n\n{text}"
)
llm    = ChatOpenAI(model="gpt-4o-mini")
parser = StrOutputParser()

# LCEL pipe operator: data flows left → right
chain = prompt | llm | parser

result = chain.invoke({
    "text": "LangChain is an open-source framework for building LLM-powered \
applications. It provides abstractions for prompts, chains, agents, \
memory, and tool use."
})
print(result)
# → "LangChain is an open-source framework that simplifies building
#    LLM-powered applications through modular abstractions."

Memory — Giving the LLM a Mind

LLMs are stateless — every call starts fresh. Memory components bridge this gap by persisting conversation history and injecting it back into future prompts. LangChain offers four main strategies:

  • ConversationBufferMemory — stores the full raw history (simple, expensive for long chats)
  • ConversationSummaryMemory — summarizes old turns to save tokens
  • ConversationEntityMemory — extracts and tracks named entities across turns
  • VectorStoreRetrieverMemory — stores history as embeddings, retrieves relevant past context
Python
# memory_example.py
from langchain_openai import ChatOpenAI
from langchain.memory import ConversationBufferMemory
from langchain.chains import ConversationChain

llm    = ChatOpenAI(model="gpt-4o-mini", temperature=0.5)
memory = ConversationBufferMemory()

conv = ConversationChain(llm=llm, memory=memory, verbose=False)

r1 = conv.predict(input="My name is Arjun.")
r2 = conv.predict(input="What's my name?")
print(r2)  # → "Your name is Arjun."

# Inspect what's stored in memory
print(memory.load_memory_variables({})["history"])

Agents — LLMs That Take Actions

An Agent is a control loop where the LLM is the brain: given a goal, it chooses a tool to call, inspects the result, then decides whether to call another tool or return a final answer. This ReAct (Reason + Act) pattern enables LLMs to solve multi-step problems dynamically.

Tools

Tools are Python callables registered with an agent. LangChain ships 50+ built-in tools (Wikipedia, DuckDuckGo, Python REPL, SQL, Wolfram Alpha) and a @tool decorator to create custom ones in seconds.

Document Loaders and Vector Stores

To build a Q&A system over your own documents, you need two things: a Document Loader to ingest content (PDFs, websites, GitHub, Notion…) and a Vector Store (FAISS, Chroma, Pinecone, Weaviate) to store embeddings for semantic retrieval. Together they form the Retrieval-Augmented Generation (RAG) pattern.

The diagram below maps the complete LangChain execution pipeline — from raw user text to a structured final answer. Memory wraps the entire loop, feeding persistent context back into future turns. The Vector Store sits as a side-channel knowledge source that can be queried at the Chain layer.

Figure 1 — LangChain end-to-end architecture. The memory loop and Vector Store are optional but critical for production applications.

The flow works as follows: raw user text is formatted by a Prompt Template, sent to an LLM, whose output is routed through a Chain. If the task requires external actions, the chain hands off to an Agent, which calls one or more Tools before producing a final answer. Memory wraps the entire loop so that context persists across turns. A Vector Store can be queried at any point to inject relevant document chunks.

Hands-on Code Examples

Agent with Tool Use

The following example creates an agent equipped with a web search tool. The LLM autonomously decides when to search, processes the result, and synthesizes a final answer — without any hard-coded if/else logic.

Python
# agent_with_tools.py — ReAct agent using DuckDuckGo search
from langchain_openai import ChatOpenAI
from langchain.agents import create_react_agent, AgentExecutor
from langchain_community.tools import DuckDuckGoSearchRun
from langchain_core.tools import tool
from langchain import hub

# 1. Define tools — built-in + custom
search = DuckDuckGoSearchRun()

@tool
def calculator(expression: str) -> str:
    """Evaluate a mathematical expression. Input must be a valid Python expression."""
    try:
        return str(eval(expression, {"__builtins__": {}}, {}))
    except Exception as e:
        return f"Error: {e}"

tools = [search, calculator]

# 2. Pull the ReAct system prompt from LangChain Hub
prompt = hub.pull("hwchase17/react")

# 3. Create agent + executor
llm   = ChatOpenAI(model="gpt-4o-mini", temperature=0)
agent = create_react_agent(llm=llm, tools=tools, prompt=prompt)
executor = AgentExecutor(agent=agent, tools=tools, verbose=True)

# 4. Run it
result = executor.invoke({
    "input": "What is today's USD to INR exchange rate? Multiply it by 1.08."
})
print(result["output"])

Full RAG Pipeline — Q&A over a PDF

The RAG (Retrieval-Augmented Generation) pattern is the backbone of most enterprise LLM applications. Documents are chunked, embedded, and stored in a vector database. At query time, relevant chunks are retrieved and injected into the prompt as context.

Python
# rag_pipeline.py — PDF Q&A with FAISS vector store
from langchain_community.document_loaders import PyPDFLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_openai import OpenAIEmbeddings, ChatOpenAI
from langchain_community.vectorstores import FAISS
from langchain.chains import RetrievalQA

# Step 1 — Load and chunk the document
loader   = PyPDFLoader("research_paper.pdf")
docs     = loader.load()

splitter = RecursiveCharacterTextSplitter(
    chunk_size=800,
    chunk_overlap=100
)
chunks   = splitter.split_documents(docs)

# Step 2 — Embed and store in FAISS (in-memory vector DB)
embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
vectordb   = FAISS.from_documents(chunks, embeddings)
retriever  = vectordb.as_retriever(search_kwargs={"k": 4})

# Step 3 — Build the Q&A chain
qa_chain = RetrievalQA.from_chain_type(
    llm=ChatOpenAI(model="gpt-4o-mini"),
    chain_type="stuff",
    retriever=retriever,
    return_source_documents=True
)

# Step 4 — Query
answer = qa_chain.invoke({"query": "What methodology was used in this study?"})
print(answer["result"])
print(f"\nSources: {[d.metadata['page'] for d in answer['source_documents']]}")
Pro Tip — Chunking Strategy
Chunk size dramatically affects RAG quality. 512–1000 tokens per chunk works well for most prose. Always set chunk_overlap=100–200 so context isn't lost at boundaries. For structured docs (tables, code), try SemanticChunker instead.

Real-World Use Cases

1. Enterprise Document Q&A (Internal Knowledge Base)

Problem: A company has 10,000+ internal PDFs, SOPs, and wiki pages. Employees waste hours searching for answers that are buried in documents.

Solution with LangChain: Ingest all documents using PyPDFLoader and UnstructuredLoader. Embed with OpenAI embeddings. Store in Pinecone or Chroma. Build a ConversationalRetrievalChain with ConversationBufferMemory so employees can ask multi-turn questions like "What does our leave policy say? What about remote work exceptions?"

2. Autonomous Research Assistant

Problem: A data analyst needs to gather competitive intelligence — pulling data from the web, summarizing articles, and computing statistics — a multi-hour manual task.

Solution with LangChain: A create_react_agent with DuckDuckGo search, a Python REPL tool, and a custom CSV writer tool. The agent is given a goal: "Research the top 5 EV manufacturers by Q4 2025 revenue. Create a comparison table." It autonomously searches, reads pages, extracts numbers, and writes a structured CSV — all with zero human-in-the-loop steps.

Problem: A SaaS company's support team handles 1,000+ repetitive tickets per day. 80% of questions are answered by the documentation.

Solution with LangChain: RAG pipeline over the docs + ConversationSummaryMemory to track issue context across a session. Advantages and Limitations

✓ Strengths

  • Massive ecosystem — 100+ integrations out of the box
  • Rapid prototyping with LCEL's expressive pipe syntax
  • Pluggable components — swap models, memory, or vector DBs with one line
  • Built-in streaming support for real-time UX
  • LangSmith for tracing, debugging, and evaluation
  • Active community and fast release cycle

✗ Limitations

  • Abstraction layers make debugging non-trivial
  • API surface changes frequently (breaking changes are common)
  • Agent loops can be unpredictable or stuck in cycles
  • Latency adds up — each LLM call is ~500ms–2s
  • Cost can escalate with multi-hop agent pipelines
  • Over-engineering risk — many tasks don't need a full agent
When NOT to use LangChain
If you're making a single, well-defined LLM call with no retrieval, memory, or tool use — just call the OpenAI API directly. LangChain adds overhead that isn't justified for simple completions. Similarly, if your pipeline has strict latency requirements (<200ms), the abstraction layers may be a liability.

Conclusion

LangChain represents a fundamental shift in how we build software. Rather than writing static programs, we compose systems that can reason, retrieve, remember, and act — with the LLM as the cognitive engine and LangChain as the nervous system connecting it to the world.

The most important insight from this deep dive is that LangChain's value is proportional to the complexity of your task. Simple tasks — a single translation, a one-shot classification — don't need it. But as soon as you need multi-turn memory, document grounding, or tool-using autonomy, the framework pays dividends immediately.

Key Takeaways

  • LCEL's pipe operator (|) is the cleanest way to compose LLM pipelines — prefer it over legacy chain classes
  • Memory and Vector Stores are what turn a toy demo into a production application
  • Agents are powerful but expensive — always set max_iterations and add a fallback
  • Chunking strategy and embedding model choice are the two biggest levers in RAG quality

Future Scope

LangGraph (LangChain's sibling framework) extends the model to stateful, cyclical workflows — enabling true multi-agent systems where specialized agents collaborate, hand off tasks, and loop back based on results. This is the frontier: autonomous AI systems that can plan, delegate, and self-correct over arbitrarily long horizons. If you've mastered the concepts in this blog, LangGraph is your logical next step.
